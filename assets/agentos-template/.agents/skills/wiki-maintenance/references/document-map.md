# Project Memory Document Map

Use this map when deciding where information belongs.

## Entry / Schema

- `wiki/AGENTS.md`: canonical shared agent rules.
- `AGENTS.md`: root symlink to `wiki/AGENTS.md`; Codex reads it.
- `CLAUDE.md`: root symlink to `wiki/AGENTS.md`; Claude reads it.

Do not put current progress, temporary plans, chat transcripts, or raw source summaries here.

## Work State

- `PLANS.md`: canonical active and recently completed plans.
- `PROGRESS.md`: canonical append-only evidence-backed timeline.
- `DECISIONS.md`: canonical durable decisions and reasons.
- `HANDOFF.md`: canonical current state for the next agent.
- `wiki/ledgers/PLANS.md`, `wiki/ledgers/PROGRESS.md`, `wiki/ledgers/DECISIONS.md`, `wiki/ledgers/HANDOFF.md`: wiki-side symlinks back to the root ledgers.
- `wiki/TASKS/`: per-task contracts with Goal, Boundaries, Evidence, Autonomy, Handoff.

These are canonical ledgers. Do not replace them with scattered wiki pages.
`wiki/` may summarize or link to them, but `DECISIONS.md` remains append-only and `PROGRESS.md` / `HANDOFF.md` stay dedicated high-update files.
They are first-class memory sources: read, query, cite, link, and promote into them when needed.

## Human Intent

- `wiki/CHATS/`: distilled chat notes that affect future work.
- Store short quotes only when exact wording matters.
- Do not save full chat logs unless requested.

## Knowledge

- `wiki/raw/`: original source material. Read-only by default.
- `wiki/raw/MANIFEST.md`: source inventory and ingest status.
- `wiki/knowledge/`: compiled project knowledge.
- `wiki/ledgers/`: wiki-side symlinks to canonical root ledgers.
- `wiki/TASKS/`: task contracts.
- `wiki/index.md`: map to memory and wiki pages.
- `wiki/log.md`: chronological operations log.
- `wiki/docs/`: project documentation intended for users or developers.
- `docs/`: root symlink to `wiki/docs/` when compatibility is useful.

Reusable methodology belongs in `wiki/knowledge/` or `wiki/docs/`: Superpowers docs, TDD/review workflows, research principles, and operating manuals.
Concrete project state generated by a methodology belongs in the canonical ledgers.

## File Type Rule

Not every file belongs in compiled wiki pages.

- Markdown/text/small structured files can be read directly.
- PDF, DOCX, XLSX, PPTX, HTML exports, and OCR-heavy images require the right parser or renderer first.
- Images, audio, video, archives, binaries, vendor dumps, dependency folders, and large generated outputs are link-only by default.
- If a source was not parsed, record that in `wiki/raw/MANIFEST.md` instead of claiming it was ingested.

## Code Graph

- GitNexus: code symbols, calls, imports, processes, impact, detect-changes.
- Wiki files can summarize architecture, but they are not proof of current code structure.
- For "what calls this", "what breaks", "what changed", or "rename safely", use GitNexus or direct source reads.

## Errors

- `wiki/errors/`: confirmed agent mistakes, repeated failures, and corrections that should change future behavior.
- `errors/`: root symlink to `wiki/errors/` when compatibility is useful.

Do not record normal user preference changes, new requirements, or exploration pivots as errors.

## Routing Tests

Ask:
- Will a future agent need this every session? -> `AGENTS.md` / `CLAUDE.md`.
- Is this the current next step? -> `HANDOFF.md`.
- Is this why we chose something? -> `DECISIONS.md`.
- Is this proof something happened? -> `PROGRESS.md`.
- Is this an active or recent plan? -> `PLANS.md`.
- Is this a task-specific contract? -> `wiki/TASKS/`.
- Is this a durable user-intent fragment from chat? -> `wiki/CHATS/`.
- Is this source material? -> `wiki/raw/`.
- Is this raw source parse status? -> `wiki/raw/MANIFEST.md`.
- Is this compiled knowledge from sources? -> `wiki/knowledge/`.
- Is this reusable methodology? -> `wiki/knowledge/` or `wiki/docs/`.
- Is this a confirmed repeated mistake? -> `wiki/errors/`.
